Ниже — полное описание пользовательской части приложения в формате, подходящем для бизнес-инициативы. Интерфейс и логика работы описаны вместе с указанием источников данных и ключевых правил.

---

Общая структура экрана

При входе в приложение открывается экранная форма, разделённая на две зоны. Левая боковая панель содержит информацию о текущем пользователе (ФИО, логин), кнопку смены пользователя и блок входа в административную панель. Центральная область представлена набором вкладок, каждая из которых отвечает за отдельный функциональный блок: «Новая проверка», «История проверок», «Нерабочие дни ВСП», «Аналитика по филиалу», «Визуализация».

---

Авторизация и начальные данные

Пользователь вводит логин. Система обращается к таблице users и по полю name находит запись сотрудника. Из этой записи извлекаются: полное имя (full_name), код филиала (name_filial), обозначение ВСП (name_vsp). Код филиала используется для поиска в таблице filials — оттуда берутся идентификатор и наименование филиала, а также признак блокировки (blocked). Обозначение ВСП сопоставляется с таблицей vsp — извлекаются идентификатор, наименование и дополнительное обозначение ВСП.

Если пользователь не найден, выводится сообщение об ошибке. Если филиал заблокирован (поле blocked имеет значение «истина»), доступ к приложению запрещается — пользователь видит предупреждение, и все дальнейшие действия блокируются.

После успешной авторизации все данные (ФИО, логин, филиал, ВСП, признак блокировки) сохраняются и используются на всех последующих шагах.

---

Вкладка «Новая проверка»

Отображаемые данные и выбор ВСП

При открытии вкладки автоматически отображаются ФИО и логин пользователя. ВСП подставлено по умолчанию из данных авторизации. Пользователь может изменить ВСП через выпадающий список, который формируется из таблицы vsp: выбираются все записи с кодом филиала пользователя и признаком активности close_vsp = false, отсортированные по наименованию. Дата проверки всегда равна текущему дню и недоступна для редактирования.

Проверки перед созданием сессии

Перед тем как разрешить создание новой проверки, система выполняет три проверки.

Первая: ищется незавершённый черновик текущего пользователя за сегодня. Просматривается таблица checklist_sessions — если существует запись, где user_name совпадает с логином пользователя, operation_date равна сегодняшней дате, а status_bul имеет значение «ложь», то черновик найден. В этом случае вместо кнопки создания проверки показываются кнопки «Продолжить» и «Удалить» для этого черновика.

Вторая: проверяется, нет ли уже любой сессии (завершённой или черновика) для выбранного ВСП на сегодняшнюю дату. В таблице checklist_sessions ищется запись, где vsp_id совпадает с выбранным ВСП, а operation_date равна текущей дате. Если такая запись существует, создание новой блокируется, так как на одно ВСП в день допускается только одна проверка.

Третья: проверяется флаг блокировки филиала, сохранённый при авторизации. Если филиал заблокирован, кнопка начала проверки недоступна.

Создание сессии

Если все проверки пройдены, при нажатии кнопки «Начать заполнение» система дополнительно проверяет, является ли сегодняшний день субботой. Если да, то автоматически удаляется запись о нерабочем дне для этого ВСП на эту дату из таблицы vsp_non_working_days (это отражает ситуацию, когда сотрудник вышел работать в свой выходной). Затем создаётся новая сессия: в таблицу checklist_sessions добавляется строка с логином пользователя, идентификатором филиала, идентификатором ВСП, текущей датой и статусом «черновик» (status_bul = false). Интерфейс переключается на форму заполнения чек-листа.

---

Форма заполнения чек-листа

Загрузка данных

После создания или продолжения черновика загружаются: шаблон проверок из таблицы checklist_templates (все пункты, отсортированные по полю item_order) и сохранённые ответы из таблицы checklist_answers для текущей сессии (по session_id). Дополнительно по филиалу сессии из таблицы filials определяется значение поля check_name. Если оно истинно, для отображения используются альтернативные поля шаблона (с префиксом alt_), иначе — стандартные.

Отображение пунктов

Для каждого пункта шаблона выводятся: порядковый номер (item_order), наименование (description), кнопка «Подробнее», раскрывающая два фильтра (основной и дополнительный). В тексте фильтров динамически подставляются: текущая дата в нескольких форматах (плейсхолдеры [Дата1], [Дата1 гггг-мм-дд], [Дата1 дд.мм.гггг]), полное название ВСП (плейсхолдер [РФ]) и первые три символа названия ВСП (плейсхолдер [РФ1]). Рядом с каждым фильтром есть кнопка копирования текста в буфер обмена.

Для каждого пункта отображается чекбокс «Выполнено» (состояние связано с полем is_completed в таблице checklist_answers) и текстовое поле «Комментарий» (связано с полем comment в той же таблице). При изменении чекбокса или комментария состояние сохраняется локально до явного сохранения.

Кнопки управления

«Сохранить черновик» — для каждого пункта, где были изменения, выполняется вставка или обновление записи в таблице checklist_answers (по паре session_id и template_item_id). Статус сессии остаётся «черновик». Сессия помечается как явно сохранённая.

«Назад» — если черновик ни разу не был явно сохранён, сессия и все её ответы удаляются (сначала из checklist_answers, затем из checklist_sessions). Если черновик сохранялся, происходит возврат к вкладке «Новая проверка» без удаления.

«Предпросмотр» — показывает количество выполненных пунктов и общее количество пунктов в шаблоне (без запросов к базе, на основе локальных данных).

«Завершить проверку» — перед кнопкой расположено необязательное текстовое поле «Комментарий к проверке». При нажатии проверяется, что все пункты отмечены. Если не все — выводится предупреждение. Если все выполнены — сохраняются ответы, сессия получает статус «завершена» (status_bul = true), фиксируется время завершения (completed_at), и общий комментарий записывается в поле completion_comment таблицы checklist_sessions. После этого интерфейс возвращается на вкладку «Новая проверка».

---

Вкладка «История проверок»

Отображает таблицу всех проверок текущего пользователя в его филиале. Данные формируются одним запросом, который объединяет таблицы checklist_sessions, filials и vsp. Для каждой сессии вычисляется количество выполненных ответов (подсчёт записей в checklist_answers с is_completed = true) и общее количество пунктов шаблона (отдельный подзапрос к checklist_templates).

Столбцы таблицы: идентификатор сессии (checklist_sessions.id), дата проверки (operation_date), филиал (filials.name), ВСП (vsp.name), статус (вычисляется: «Завершена», если status_bul истинен, иначе «Черновик»), дата и время начала (created_at), дата и время завершения (completed_at, может быть пустым для черновиков), количество выполненных проверок, общее количество проверок.

При выборе сессии и нажатии «Показать результаты» загружаются полные данные: сама сессия, все её ответы с отметками о выполнении и комментариями, а также шаблон чек-листа. Результаты выводятся в виде списка пунктов с указанием статуса каждого.

Удаление черновика: пользователь вводит идентификатор сессии; система проверяет, что сессия принадлежит ему и имеет статус «Черновик». При подтверждении удаляются все ответы и сама сессия.

---

Вкладка «Нерабочие дни ВСП»

Добавление нерабочего дня

ВСП автоматически подставлено из данных пользователя, но может быть изменено через список активных ВСП филиала (запрос к vsp с фильтром по filial_id и close_vsp = false). Пользователь выбирает дату из календаря и причину из выпадающего списка (ремонтные работы, праздничный день и т.д.). Если выбрана причина «Другое», появляется дополнительное текстовое поле для произвольного описания. При сохранении в таблицу vsp_non_working_days добавляется запись с логином пользователя, кодом филиала, идентификатором ВСП, датой и причиной.

Просмотр ранее добавленных дней

Записи извлекаются из vsp_non_working_days с присоединением справочников filials и vsp для получения читаемых названий. По умолчанию показываются только записи для выбранного ВСП. При установке флажка «Показать все выходные вашего ВСП» отображаются все записи этого ВСП (фильтр по vsp_id сохраняется). Для каждой записи выводится кнопка «Удалить». При удалении проверяется, что запись относится к тому же ВСП и филиалу, что и текущий пользователь.

---

Вкладка «Аналитика по филиалу»

Филиал автоматически определён. Пользователь может выбрать период дат через два календаря.

Таблица всех проверок строится запросом, аналогичным истории, но без фильтрации по пользователю — показываются все сессии филиала. Столбцы: идентификатор сессии, сотрудник (поле user_name), дата, ВСП, статус, дата и время начала, дата и время завершения, количество выполненных проверок, комментарий (из поля completion_comment, если заполнен).

В блоке управления черновиками отображаются только сессии со статусом «Черновик», дополнительно отфильтрованные по ВСП текущего пользователя. Пользователь может выбрать несколько черновиков и удалить их, предварительно подтвердив действие. При удалении для каждой выбранной сессии каскадно удаляются ответы и сама сессия.

---

Вкладка «Визуализация»

Показывает статус заполнения чек-листов для всех ВСП филиала на выбранную дату. Пользователь видит свой филиал (автоматически определён) и выбирает дату.

При нажатии «Показать статус» система получает: перечень всех активных ВСП филиала (из vsp), множество ВСП с нерабочим днём на выбранную дату (из vsp_non_working_days) и множество ВСП с завершённой проверкой на эту дату (из checklist_sessions с условием status_bul = true). Для каждого ВСП определяется статус: если ВСП есть в списке нерабочих — «Выходной»; иначе если есть завершённая проверка — «Заполнен»; иначе — «Не заполнен».

Строится круговая диаграмма, где ВСП со статусом «Выходной» исключаются из расчёта долей. Оставшиеся делятся на «Заполнено» (зелёный) и «Не заполнено» (красный) с процентным соотношением. Под диаграммой — таблица со столбцами «ВСП» и «Статус», строки которой раскрашены в зависимости от статуса.

---

Ключевые бизнес-правила пользовательской части

На один ВСП в день может быть создана только одна проверка (независимо от того, завершена она или является черновиком). Один пользователь не может иметь более одного незавершённого черновика за текущую дату. Если филиал пользователя заблокирован (поле blocked в таблице filials), создание новых проверок полностью запрещено. Если проверка создаётся в субботу, ранее установленный для этого ВСП нерабочий день автоматически удаляется. Выбор между стандартными и альтернативными фильтрами при заполнении чек-листа зависит от настройки филиала (поле check_name в таблице filials). Удаление черновика разрешено только его владельцу; при удалении каскадно удаляются все связанные ответы.

---

Вкладка «Новая проверка»

Информация о пользователе и ВСП

· ФИО: из users.full_name по логину.
· Логин: из users.name.
· ВСП по умолчанию: vsp.name (и vsp.name_vsp) через связь users.name_vsp = vsp.name (для определения ВСП пользователя).
· Филиал: filials.name через users.name_filial::numeric = filials.id.
· Блокировка филиала: filials.blocked (если true – доступ запрещён).

Выбор ВСП

· Список ВСП: vsp.name, vsp.name_vsp для vsp.filial_id = :filial_id AND close_vsp = false, сортировка по name.

Проверки перед началом

· Наличие черновика за сегодня: запрос к checklist_sessions с условиями user_name = :current_user AND operation_date = today AND status_bul = false.
· Существующая проверка (любая) для ВСП на дату: запрос к checklist_sessions с vsp_id = :vsp_id AND operation_date = :date.

Создание сессии

· Вставка в checklist_sessions (user_name, filial_id, vsp_id, operation_date, status_bul=false).
· При субботе: удаление нерабочего дня (DELETE FROM vsp_non_working_days WHERE vsp_id=:vsp_id AND date=:date).

---

Вкладка «История проверок»

Таблица истории

Запрос объединяет сессии с филиалами, ВСП и подсчитывает ответы.

```sql
SELECT 
    s.id,
    s.operation_date AS "Дата проверки",
    f.name AS "Филиал",
    v.name AS "ВСП",
    CASE s.status_bul WHEN true THEN 'Завершена' ELSE 'Черновик' END AS "Статус",
    s.created_at AS "Дата и время начала",
    s.completed_at AS "Дата и время завершения",
    COUNT(a.id) AS "Выполнено проверок",
    (SELECT COUNT(*) FROM checklist_templates) AS "Всего проверок"
FROM checklist_sessions s
JOIN filials f ON s.filial_id = f.id
JOIN vsp v ON s.vsp_id = v.id
LEFT JOIN checklist_answers a ON s.id = a.session_id AND a.is_completed = true
WHERE s.user_name = :current_user AND s.filial_id = :filial_id
GROUP BY s.id, f.name, v.name, ...
ORDER BY s.created_at DESC
```

· ID сессии: s.id.
· Дата проверки: s.operation_date.
· Филиал: f.name.
· ВСП: v.name.
· Статус: вычисляется из s.status_bul.
· Дата и время начала: s.created_at.
· Дата и время завершения: s.completed_at.
· Выполнено проверок: COUNT(a.id) WHERE is_completed = true.
· Всего проверок: общее количество пунктов в checklist_templates (подзапрос).

Детальный просмотр сессии

· Загружается сама сессия (SELECT * FROM checklist_sessions WHERE id = :session_id).
· Загружаются ответы: SELECT template_item_id, is_completed, comment FROM checklist_answers WHERE session_id = :session_id.
· Шаблон чек-листа (SELECT * FROM checklist_templates ORDER BY item_order).
· При отображении: для каждого пункта показывается статус (is_completed) и описание из шаблона.

Удаление черновика

· Удаление из checklist_answers и checklist_sessions по ID (каскадное).

---

Вкладка «Нерабочие дни ВСП»

Добавление нерабочего дня

· Поля: ВСП (из vsp), дата, причина.
· Сохранение: INSERT INTO vsp_non_working_days (user_name, filial_id, vsp_id, date, reason).

Ранее добавленные нерабочие дни

· Запрос с JOIN к filials и vsp:

```sql
SELECT 
    nwd.id, nwd.user_name, f.name AS filial, v.name AS vsp,
    nwd.date, nwd.reason, nwd.created_at
FROM vsp_non_working_days nwd
JOIN filials f ON nwd.filial_id = f.id
JOIN vsp v ON nwd.vsp_id = v.id
WHERE nwd.filial_id = :filial_id [AND nwd.vsp_id = :vsp_id]
ORDER BY nwd.date DESC, nwd.created_at DESC
```

· Возможность фильтрации по конкретному ВСП или все (чекбокс «Показать все выходные вашего ВСП» управляет наличием условия vsp_id = :vsp_id).
· Удаление: DELETE FROM vsp_non_working_days WHERE id = :id AND vsp_id = :vsp_id AND filial_id = :filial_id.

Вкладка «Аналитика по филиалу»

Фильтры

· Филиал берётся автоматически из last_filial_id пользователя.
· Период дат (date_from, date_to).

Таблица всех проверок

· Используется запрос get_filial_sessions(filial_id) с дополнительной фильтрацией по дате.
· Поля аналогичны истории, но для всех пользователей филиала:
  · id, Сотрудник (s.user_name), Дата проверки, ВСП, Статус, Выполнено проверок, Всего проверок, Дата и время начала, Дата и время завершения.
  · Если реализован комментарий к сессии, добавляется s.completion_comment.

Управление черновиками

· Черновики = status_bul = false.
· Дополнительный фильтр по ВСП = last_vsp_name пользователя.
· Отображение: ID, Сотрудник, Дата, Прогресс.
· Массовое удаление: цикл по выбранным ID с вызовом delete_session.

Вкладка «Визуализация»

Статус ВСП на дату

· Источник: объединение данных из vsp (активные), vsp_non_working_days и checklist_sessions.
· Логика:
  1. Получить все активные ВСП филиала (vsp WHERE filial_id = :filial_id AND close_vsp = false).
