



Ниже — полное описание административной части приложения в формате, подходящем для бизнес-инициативы. Интерфейс и логика работы описаны вместе с указанием источников данных и ключевых правил.

---

Доступ к административной панели

Административная панель доступна после проверки прав. Возможны два способа входа. Первый — через корпоративную систему аутентификации Keycloak: если email пользователя содержится в заранее определённом списке администраторов (хранится в конфигурации приложения), вход в режим администратора происходит автоматически при установке флажка на боковой панели. Второй — через ввод пароля: если вход через Keycloak не используется или email отсутствует в списке, администратор вводит пароль в текстовое поле. Пароль сравнивается с заданным значением, хранящимся в конфигурации приложения. При совпадении режим администратора активируется.

После успешной активации на боковой панели появляются дополнительные элементы управления, а в центральной области добавляются новые вкладки: «Аналитика», «Филиалы (чекбоксы)», «Нерабочие дни (отчет)», «Визуализация (админ)», «Витрины (админ)».

---

Левая боковая панель (административные функции)

Просмотр текущих пунктов шаблона

В свёрнутом блоке отображается список всех пунктов чек-листа. Данные загружаются из таблицы checklist_templates (поля item_order и description), сортировка по порядковому номеру. Для каждого пункта показывается его номер и наименование.

Добавление новой проверки в шаблон

В расширенном блоке администратор заполняет текстовые поля: наименование проверки, описание, основной фильтр, дополнительный фильтр, альтернативный фильтр, альтернативное описание, альтернативный дополнительный фильтр. Все эти данные будут сохранены в соответствующих полях таблицы checklist_templates (description, additional_info, filter_value, events_value, alt_filter_value, alt_additional_info, alt_events_value).

Также можно выбрать позицию вставки: либо в конец списка, либо перед конкретным существующим пунктом. Если выбрана позиция не в конец, все существующие пункты с порядковым номером, большим или равным указанному, сдвигаются на единицу вперёд, после чего новый пункт вставляется на освободившееся место. Это позволяет управлять порядком проверок без ручной перенумерации.

Редактирование пункта шаблона

Администратор выбирает пункт из выпадающего списка, где для каждого элемента показан его идентификатор и первые символы наименования. При выборе загружаются текущие значения всех полей из таблицы checklist_templates. Администратор может изменить любое поле и сохранить изменения. Выполняется обновление записи в таблице по идентификатору пункта.

Удаление пункта шаблона

При удалении выполняются три операции. Сначала удаляются все ответы, связанные с этим пунктом, из таблицы checklist_answers (по полю template_item_id). Затем удаляется сам пункт из таблицы checklist_templates. После этого происходит автоматическая перенумерация оставшихся пунктов: каждому присваивается новый порядковый номер на основе текущей сортировки, чтобы последовательность оставалась непрерывной.

Перемещение пункта шаблона

Поддерживается два способа перемещения. Первый — сдвиг вверх или вниз на одну позицию: система находит соседний пункт (с предыдущим или следующим порядковым номером) и меняет их номера местами. Второй — перемещение на конкретную позицию: в зависимости от направления перемещения все промежуточные пункты сдвигаются на единицу, а перемещаемый пункт получает новый номер. Оба способа работают через обновление поля item_order в таблице checklist_templates.

Обновление ФИО пользователя

В скрытом блоке администратор может изменить полное имя сотрудника. Он вводит логин пользователя (поле name в таблице users). Система проверяет существование пользователя и отображает его текущее ФИО (поле full_name). Затем администратор вводит новое ФИО и сохраняет изменение. Выполняется обновление поля full_name в таблице users по введённому логину.

Вкладка «Аналитика» (административная)

Фильтры

В верхней части вкладки расположены четыре элемента фильтрации. Филиал выбирается из выпадающего списка, сформированного из таблицы filials (поля id и name), с дополнительной опцией «Все». Список ВСП зависит от выбранного филиала: если выбран конкретный филиал, показываются активные ВСП этого филиала (из таблицы vsp, условие filial_id и close_vsp = false); если выбрано «Все», показываются все активные ВСП. Также доступна опция «Все». Период задаётся двумя календарями — «Дата от» и «Дата до». Все фильтры сохраняются и синхронизируются между административными вкладками.

Таблица проверок с детализацией

При нажатии кнопки «Показать данные» система извлекает сессии из таблицы checklist_sessions с присоединением справочников filials и vsp для получения названий. Затем одним пакетным запросом загружаются все ответы для найденных сессий из таблицы checklist_answers. Отдельно загружается шаблон из checklist_templates. Для каждого пункта шаблона в итоговой таблице создаётся отдельный столбец, показывающий, выполнен ли этот пункт в данной сессии (значение формируется на основе наличия записи в checklist_answers с is_completed = true). В таблице также присутствуют столбцы: ФИО сотрудника, Филиал, ВСП, Дата, Статус (завершена или черновик), Комментарий (из поля completion_comment сессии), Дата и время завершения.

Возврат завершённой сессии в черновик

Администратор вводит часть ФИО сотрудника. Система ищет все завершённые сессии (поле status_bul имеет значение «истина»), где ФИО соответствует введённому шаблону, и отображает их списком. Для каждой найденной сессии показывается: идентификатор, ФИО, филиал, ВСП, дата операции, дата завершения. Администратор выбирает нужную сессию и нажимает «Вернуть в черновик». При этом в таблице checklist_sessions для выбранной записи поле status_bul устанавливается в «ложь», а поле completed_at очищается. После этого сессия снова доступна для продолжения заполнения.

Удаление черновиков старше 24 часов

Система автоматически находит все сессии, где status_bul имеет значение «ложь», а дата создания (created_at) старше текущего момента более чем на 24 часа. Отображается количество найденных черновиков и таблица с их списком. Администратор должен подтвердить удаление, установив специальный флажок. После подтверждения для каждой найденной сессии каскадно удаляются все связанные ответы (из таблицы checklist_answers) и сама сессия (из таблицы checklist_sessions).

Удаление черновиков по ФИО

Администратор вводит часть ФИО сотрудника. Система находит все черновики (сессии с status_bul равным «ложь»), где ФИО соответствует запросу. Результаты отображаются в редактируемой таблице, где администратор может отметить галочками отдельные строки для удаления. При подтверждении для каждой отмеченной сессии выполняется каскадное удаление ответов и самой сессии.

Вкладка «Филиалы (чекбоксы)»

Отображает список всех филиалов из таблицы filials (поля id, name, check_name, blocked). Для каждого филиала выводятся его название и два чекбокса.

Первый чекбокс — «Альтернативный фильтр». Он управляет значением поля check_name в таблице filials. Когда флажок установлен, при заполнении чек-листа пользователями этого филиала будут использоваться альтернативные описания и фильтры из шаблона (поля с префиксом alt_ в таблице checklist_templates). Когда флажок снят, используются стандартные.

Второй чекбокс — «Блокировка». Он управляет значением поля blocked в таблице filials. Когда флажок установлен, пользователи этого филиала полностью лишаются возможности создавать новые проверки — при попытке они видят сообщение о временном ограничении доступа.

При нажатии кнопки «Сохранить настройки» для каждого филиала выполняется обновление обоих полей в таблице filials.

---

Вкладка «Нерабочие дни (отчет)»

Фильтры и просмотр

Фильтры аналогичны вкладке «Аналитика»: выбор филиала, ВСП, периода дат. При нажатии кнопки «Показать» данные извлекаются из таблицы vsp_non_working_days с присоединением справочников filials и vsp для получения читаемых названий. Результат отображается в редактируемой таблице с колонками: идентификатор записи, филиал, ВСП, дата, причина. Администратор может отметить несколько записей для удаления. При подтверждении все отмеченные записи удаляются из таблицы vsp_non_working_days по их идентификаторам.

Массовое добавление для всех ВСП филиала

Администратор выбирает филиал, дату и причину. При выполнении система загружает все активные ВСП выбранного филиала (из таблицы vsp, условие filial_id и close_vsp = false). Для каждого ВСП проверяется, существует ли уже запись о нерабочем дне на указанную дату (поиск в vsp_non_working_days по vsp_id и date). Если записи нет, создаётся новая с указанием администратора в качестве автора, кодом филиала, идентификатором ВСП, датой и причиной. Система возвращает статистику: сколько записей добавлено и сколько пропущено из-за того, что они уже существовали.

Штучное добавление (один день или период)

Администратор выбирает филиал, конкретное ВСП, тип добавления («Один день» или «Период»), дату или диапазон дат и причину. При выборе одного дня выполняется одиночная вставка с предварительной проверкой существования. При выборе периода система проходит по всем датам в указанном диапазоне и для каждой выполняет проверку существования записи для данного ВСП. Если записи нет, создаётся новая. Возвращается статистика по добавленным и пропущенным дням.

Добавление выходных на все субботы

Администратор указывает диапазон дат. Система перебирает все даты в этом диапазоне. Для каждой даты, которая является субботой, и для каждого активного ВСП (из таблицы vsp, без фильтра по филиалу) проверяется существование записи о нерабочем дне. Если записи нет, создаётся новая с причиной «Суббота (выходной)». Возвращается общая статистика по добавленным и пропущенным записям.

Массовое удаление суббот для нескольких ВСП

Администратор выбирает филиал, затем одно или несколько ВСП из списка (мультивыбор). Также указывается период дат. При выполнении из таблицы vsp_non_working_days удаляются все записи, где идентификатор ВСП входит в выбранный список, дата находится в указанном диапазоне, и день недели является субботой (проверка через функцию определения дня недели). Записи для других дней недели не затрагиваются. Возвращается количество удалённых записей.

Вкладка «Визуализация (админ)»

Статус ВСП выбранного филиала

Администратор выбирает филиал из выпадающего списка (сформированного из таблицы filials) и дату. При нажатии «Показать статус» выполняются три запроса. Первый получает все активные ВСП выбранного филиала (из таблицы vsp, условие filial_id и close_vsp = false). Второй — список ВСП, для которых на выбранную дату зарегистрирован нерабочий день (из таблицы vsp_non_working_days). Третий — список ВСП, для которых на эту дату существует завершённая проверка (из таблицы checklist_sessions, условие status_bul = true).

Для каждого ВСП вычисляется статус: если ВСП есть в списке нерабочих — «Выходной»; иначе если есть завершённая проверка — «Заполнен»; иначе — «Не заполнен». Строится круговая диаграмма, где ВСП со статусом «Выходной» исключаются, а оставшиеся делятся на заполненные и незаполненные с цветовой индикацией (зелёный и красный) и процентным соотношением. Под диаграммой отображается таблица со столбцами «ВСП» и «Статус», строки которой раскрашены в зависимости от статуса.

Сводка по всем филиалам

Дополнительно для этой же даты строится сводный статус по всем филиалам. Для каждого филиала система определяет количество активных ВСП, количество ВСП с нерабочим днём и количество ВСП с завершённой проверкой. На основе этих данных вычисляется общий статус филиала: если все активные ВСП — нерабочие, статус «Все выходные»; если все активные ВСП (исключая нерабочие) имеют завершённую проверку — «Заполнен»; если есть хотя бы одна завершённая проверка, но не все — «Частично»; если нет ни одной завершённой проверки — «Не заполнен».

Результаты выводятся в виде круговой диаграммы по статусам филиалов и таблицы с колонками: Филиал, Статус, Всего ВСП, Активных ВСП (исключая нерабочие), Заполнено ВСП. Строки таблицы раскрашены в зависимости от статуса.

Вкладка «Витрины (админ)»

Содержит четыре аналитических отчёта, использующих сложные запросы с общими табличными выражениями и объединения с календарём рабочих дней.

Витрина «Завершение операций РФ/ВСП»

Сводный отчёт по состоянию исполнения контролей за каждый день. Объединяет данные о завершённых проверках (из таблицы checklist_sessions, только записи со статусом «завершена») и ошибках, выявленных в смежной системе (из таблицы checklist_rf.incomplete_checks_czo). Для каждого ВСП на каждую дату из календаря определяется: был ли факт завершения проверки и были ли зафиксированы ошибки. Результат — таблица с колонками: РФ (первые три символа названия ВСП из vsp.name), ВСП (полное название), Дата операционного дня, Результат (символ, показывающий, пройдена ли проверка), Контроль (показывает наличие ошибок по данным смежной системы). Также формируется сводный столбец, объединяющий оба показателя.

Отчёт «Завершение операций по РФ»

Сводный за месяц рейтинг по всем филиалам в двух измерениях. Первое измерение — доля ВСП, в которых были допущены ошибки (от общего количества активных ВСП филиала, исключая дни, когда ВСП не работали). Второе измерение — доля операционных дней с ошибками (от общего количества рабочих дней). Для каждого филиала и месяца вычисляются оба показателя и присваиваются рейтинги. Результат — таблица с колонками: Год, Месяц, РФ, «Количество ВСП с ошибками» (в формате «с ошибками / всего»), Рейтинг по ВСП, «Количество дней с ошибками» (в формате «с ошибками / всего рабочих дней»), Рейтинг по дням.

Отчёт «Завершение операций по ВСП»

Аналогичен предыдущему, но детализация выполняется до уровня отдельных ВСП. Для каждого ВСП и месяца считается количество дней с ошибками и общее количество рабочих дней, после чего вычисляется рейтинг. Результат — таблица с колонками: Год, Месяц, ВСП, «Количество дней с ошибками», Рейтинг.

Детальный отчёт по завершению операций

Реестр конкретных контрольных процедур, по которым были допущены ошибки. Для каждой ошибки, зафиксированной в таблице checklist_rf.incomplete_checks_czo со значением «истина», находится соответствующий контроль из шаблона checklist_templates (по идентификатору шаблона) и завершённая проверка из checklist_sessions (по ВСП и дате), чтобы установить исполнителя. Результат — таблица с колонками: РФ (код филиала), ВСП (название), Дата операционного дня, Исполнитель (поле user_name из сессии), Контроль (описание из шаблона с порядковым номером).

---

Ключевые бизнес-правила административной части

При удалении пункта шаблона каскадно удаляются все связанные с ним ответы, после чего оставшиеся пункты автоматически перенумеровываются для сохранения непрерывной последовательности. При массовых операциях с нерабочими днями перед созданием каждой записи проверяется её отсутствие для того же ВСП на ту же дату — дубликаты не создаются. При удалении субботних нерабочих дней фильтр по дню недели гарантирует, что затрагиваются только субботы; записи для других дней, включая воскресенья и будние дни, остаются нетронутыми. Установка блокировки филиала немедленно запрещает пользователям этого филиала создавать новые проверки, но не затрагивает уже существующие черновики. Флаг альтернативных фильтров влияет на то, какие поля шаблона увидят пользователи филиала при заполнении чек-листа. При возврате завершённой сессии в черновик у неё очищается дата завершения и меняется статус, что делает её снова доступной для редактирования.

Общая структура экрана

При входе в приложение открывается экранная форма, разделённая на две зоны. Левая боковая панель содержит информацию о текущем пользователе (ФИО, логин), кнопку смены пользователя и блок входа в административную панель. Центральная область представлена набором вкладок, каждая из которых отвечает за отдельный функциональный блок: «Новая проверка», «История проверок», «Нерабочие дни ВСП», «Аналитика по филиалу», «Визуализация».

---

Авторизация и начальные данные

Пользователь вводит логин. Система обращается к таблице users и по полю name находит запись сотрудника. Из этой записи извлекаются: полное имя (full_name), код филиала (name_filial), обозначение ВСП (name_vsp). Код филиала используется для поиска в таблице filials — оттуда берутся идентификатор и наименование филиала, а также признак блокировки (blocked). Обозначение ВСП сопоставляется с таблицей vsp — извлекаются идентификатор, наименование и дополнительное обозначение ВСП.

Если пользователь не найден, выводится сообщение об ошибке. Если филиал заблокирован (поле blocked имеет значение «истина»), доступ к приложению запрещается — пользователь видит предупреждение, и все дальнейшие действия блокируются.

После успешной авторизации все данные (ФИО, логин, филиал, ВСП, признак блокировки) сохраняются и используются на всех последующих шагах.

---

Вкладка «Новая проверка»

Отображаемые данные и выбор ВСП

При открытии вкладки автоматически отображаются ФИО и логин пользователя. ВСП подставлено по умолчанию из данных авторизации. Пользователь может изменить ВСП через выпадающий список, который формируется из таблицы vsp: выбираются все записи с кодом филиала пользователя и признаком активности close_vsp = false, отсортированные по наименованию. Дата проверки всегда равна текущему дню и недоступна для редактирования.

Проверки перед созданием сессии

Перед тем как разрешить создание новой проверки, система выполняет три проверки.

Первая: ищется незавершённый черновик текущего пользователя за сегодня. Просматривается таблица checklist_sessions — если существует запись, где user_name совпадает с логином пользователя, operation_date равна сегодняшней дате, а status_bul имеет значение «ложь», то черновик найден. В этом случае вместо кнопки создания проверки показываются кнопки «Продолжить» и «Удалить» для этого черновика.

Вторая: проверяется, нет ли уже любой сессии (завершённой или черновика) для выбранного ВСП на сегодняшнюю дату. В таблице checklist_sessions ищется запись, где vsp_id совпадает с выбранным ВСП, а operation_date равна текущей дате. Если такая запись существует, создание новой блокируется, так как на одно ВСП в день допускается только одна проверка.

Третья: проверяется флаг блокировки филиала, сохранённый при авторизации. Если филиал заблокирован, кнопка начала проверки недоступна.

Создание сессии

Если все проверки пройдены, при нажатии кнопки «Начать заполнение» система дополнительно проверяет, является ли сегодняшний день субботой. Если да, то автоматически удаляется запись о нерабочем дне для этого ВСП на эту дату из таблицы vsp_non_working_days (это отражает ситуацию, когда сотрудник вышел работать в свой выходной). Затем создаётся новая сессия: в таблицу checklist_sessions добавляется строка с логином пользователя, идентификатором филиала, идентификатором ВСП, текущей датой и статусом «черновик» (status_bul = false). Интерфейс переключается на форму заполнения чек-листа.

---

Форма заполнения чек-листа

Загрузка данных

После создания или продолжения черновика загружаются: шаблон проверок из таблицы checklist_templates (все пункты, отсортированные по полю item_order) и сохранённые ответы из таблицы checklist_answers для текущей сессии (по session_id). Дополнительно по филиалу сессии из таблицы filials определяется значение поля check_name. Если оно истинно, для отображения используются альтернативные поля шаблона (с префиксом alt_), иначе — стандартные.

Отображение пунктов

Для каждого пункта шаблона выводятся: порядковый номер (item_order), наименование (description), кнопка «Подробнее», раскрывающая два фильтра (основной и дополнительный). В тексте фильтров динамически подставляются: текущая дата в нескольких форматах (плейсхолдеры [Дата1], [Дата1 гггг-мм-дд], [Дата1 дд.мм.гггг]), полное название ВСП (плейсхолдер [РФ]) и первые три символа названия ВСП (плейсхолдер [РФ1]). Рядом с каждым фильтром есть кнопка копирования текста в буфер обмена.

Для каждого пункта отображается чекбокс «Выполнено» (состояние связано с полем is_completed в таблице checklist_answers) и текстовое поле «Комментарий» (связано с полем comment в той же таблице). При изменении чекбокса или комментария состояние сохраняется локально до явного сохранения.

Кнопки управления

«Сохранить черновик» — для каждого пункта, где были изменения, выполняется вставка или обновление записи в таблице checklist_answers (по паре session_id и template_item_id). Статус сессии остаётся «черновик». Сессия помечается как явно сохранённая.

«Назад» — если черновик ни разу не был явно сохранён, сессия и все её ответы удаляются (сначала из checklist_answers, затем из checklist_sessions). Если черновик сохранялся, происходит возврат к вкладке «Новая проверка» без удаления.

«Предпросмотр» — показывает количество выполненных пунктов и общее количество пунктов в шаблоне (без запросов к базе, на основе локальных данных).

«Завершить проверку» — перед кнопкой расположено необязательное текстовое поле «Комментарий к проверке». При нажатии проверяется, что все пункты отмечены. Если не все — выводится предупреждение. Если все выполнены — сохраняются ответы, сессия получает статус «завершена» (status_bul = true), фиксируется время завершения (completed_at), и общий комментарий записывается в поле completion_comment таблицы checklist_sessions. После этого интерфейс возвращается на вкладку «Новая проверка».

---

Вкладка «История проверок»

Отображает таблицу всех проверок текущего пользователя в его филиале. Данные формируются одним запросом, который объединяет таблицы checklist_sessions, filials и vsp. Для каждой сессии вычисляется количество выполненных ответов (подсчёт записей в checklist_answers с is_completed = true) и общее количество пунктов шаблона (отдельный подзапрос к checklist_templates).

Столбцы таблицы: идентификатор сессии (checklist_sessions.id), дата проверки (operation_date), филиал (filials.name), ВСП (vsp.name), статус (вычисляется: «Завершена», если status_bul истинен, иначе «Черновик»), дата и время начала (created_at), дата и время завершения (completed_at, может быть пустым для черновиков), количество выполненных проверок, общее количество проверок.

При выборе сессии и нажатии «Показать результаты» загружаются полные данные: сама сессия, все её ответы с отметками о выполнении и комментариями, а также шаблон чек-листа. Результаты выводятся в виде списка пунктов с указанием статуса каждого.

Удаление черновика: пользователь вводит идентификатор сессии; система проверяет, что сессия принадлежит ему и имеет статус «Черновик». При подтверждении удаляются все ответы и сама сессия.

---

Вкладка «Нерабочие дни ВСП»

Добавление нерабочего дня

ВСП автоматически подставлено из данных пользователя, но может быть изменено через список активных ВСП филиала (запрос к vsp с фильтром по filial_id и close_vsp = false). Пользователь выбирает дату из календаря и причину из выпадающего списка (ремонтные работы, праздничный день и т.д.). Если выбрана причина «Другое», появляется дополнительное текстовое поле для произвольного описания. При сохранении в таблицу vsp_non_working_days добавляется запись с логином пользователя, кодом филиала, идентификатором ВСП, датой и причиной.

Просмотр ранее добавленных дней

Записи извлекаются из vsp_non_working_days с присоединением справочников filials и vsp для получения читаемых названий. По умолчанию показываются только записи для выбранного ВСП. При установке флажка «Показать все выходные вашего ВСП» отображаются все записи этого ВСП (фильтр по vsp_id сохраняется). Для каждой записи выводится кнопка «Удалить». При удалении проверяется, что запись относится к тому же ВСП и филиалу, что и текущий пользователь.

---

Вкладка «Аналитика по филиалу»

Филиал автоматически определён. Пользователь может выбрать период дат через два календаря.

Таблица всех проверок строится запросом, аналогичным истории, но без фильтрации по пользователю — показываются все сессии филиала. Столбцы: идентификатор сессии, сотрудник (поле user_name), дата, ВСП, статус, дата и время начала, дата и время завершения, количество выполненных проверок, комментарий (из поля completion_comment, если заполнен).

В блоке управления черновиками отображаются только сессии со статусом «Черновик», дополнительно отфильтрованные по ВСП текущего пользователя. Пользователь может выбрать несколько черновиков и удалить их, предварительно подтвердив действие. При удалении для каждой выбранной сессии каскадно удаляются ответы и сама сессия.

---

Вкладка «Визуализация»

Показывает статус заполнения чек-листов для всех ВСП филиала на выбранную дату. Пользователь видит свой филиал (автоматически определён) и выбирает дату.

При нажатии «Показать статус» система получает: перечень всех активных ВСП филиала (из vsp), множество ВСП с нерабочим днём на выбранную дату (из vsp_non_working_days) и множество ВСП с завершённой проверкой на эту дату (из checklist_sessions с условием status_bul = true). Для каждого ВСП определяется статус: если ВСП есть в списке нерабочих — «Выходной»; иначе если есть завершённая проверка — «Заполнен»; иначе — «Не заполнен».

Строится круговая диаграмма, где ВСП со статусом «Выходной» исключаются из расчёта долей. Оставшиеся делятся на «Заполнено» (зелёный) и «Не заполнено» (красный) с процентным соотношением. Под диаграммой — таблица со столбцами «ВСП» и «Статус», строки которой раскрашены в зависимости от статуса.

---

Ключевые бизнес-правила пользовательской части

На один ВСП в день может быть создана только одна проверка (независимо от того, завершена она или является черновиком). Один пользователь не может иметь более одного незавершённого черновика за текущую дату. Если филиал пользователя заблокирован (поле blocked в таблице filials), создание новых проверок полностью запрещено. Если проверка создаётся в субботу, ранее установленный для этого ВСП нерабочий день автоматически удаляется. Выбор между стандартными и альтернативными фильтрами при заполнении чек-листа зависит от настройки филиала (поле check_name в таблице filials). Удаление черновика разрешено только его владельцу; при удалении каскадно удаляются все связанные ответы.

---

Это описание покрывает пользовательскую часть приложения с необходимой для бизнес-инициативы детализацией: что видит пользователь, откуда берутся данные и какие правила применяются. Административная часть будет описана отдельно по тому же принципу.

Вкладка «Новая проверка»

Информация о пользователе и ВСП

· ФИО: из users.full_name по логину.
· Логин: из users.name.
· ВСП по умолчанию: vsp.name (и vsp.name_vsp) через связь users.name_vsp = vsp.name (для определения ВСП пользователя).
· Филиал: filials.name через users.name_filial::numeric = filials.id.
· Блокировка филиала: filials.blocked (если true – доступ запрещён).

Выбор ВСП

· Список ВСП: vsp.name, vsp.name_vsp для vsp.filial_id = :filial_id AND close_vsp = false, сортировка по name.

Проверки перед началом

· Наличие черновика за сегодня: запрос к checklist_sessions с условиями user_name = :current_user AND operation_date = today AND status_bul = false.
· Существующая проверка (любая) для ВСП на дату: запрос к checklist_sessions с vsp_id = :vsp_id AND operation_date = :date.

Создание сессии

· Вставка в checklist_sessions (user_name, filial_id, vsp_id, operation_date, status_bul=false).
· При субботе: удаление нерабочего дня (DELETE FROM vsp_non_working_days WHERE vsp_id=:vsp_id AND date=:date).

---

Вкладка «История проверок»

Таблица истории

Запрос объединяет сессии с филиалами, ВСП и подсчитывает ответы.

```sql
SELECT 
    s.id,
    s.operation_date AS "Дата проверки",
    f.name AS "Филиал",
    v.name AS "ВСП",
    CASE s.status_bul WHEN true THEN 'Завершена' ELSE 'Черновик' END AS "Статус",
    s.created_at AS "Дата и время начала",
    s.completed_at AS "Дата и время завершения",
    COUNT(a.id) AS "Выполнено проверок",
    (SELECT COUNT(*) FROM checklist_templates) AS "Всего проверок"
FROM checklist_sessions s
JOIN filials f ON s.filial_id = f.id
JOIN vsp v ON s.vsp_id = v.id
LEFT JOIN checklist_answers a ON s.id = a.session_id AND a.is_completed = true
WHERE s.user_name = :current_user AND s.filial_id = :filial_id
GROUP BY s.id, f.name, v.name, ...
ORDER BY s.created_at DESC
```

· ID сессии: s.id.
· Дата проверки: s.operation_date.
· Филиал: f.name.
· ВСП: v.name.
· Статус: вычисляется из s.status_bul.
· Дата и время начала: s.created_at.
· Дата и время завершения: s.completed_at.
· Выполнено проверок: COUNT(a.id) WHERE is_completed = true.
· Всего проверок: общее количество пунктов в checklist_templates (подзапрос).

Детальный просмотр сессии

· Загружается сама сессия (SELECT * FROM checklist_sessions WHERE id = :session_id).
· Загружаются ответы: SELECT template_item_id, is_completed, comment FROM checklist_answers WHERE session_id = :session_id.
· Шаблон чек-листа (SELECT * FROM checklist_templates ORDER BY item_order).
· При отображении: для каждого пункта показывается статус (is_completed) и описание из шаблона.

Удаление черновика

· Удаление из checklist_answers и checklist_sessions по ID (каскадное).

---

Вкладка «Нерабочие дни ВСП»

Добавление нерабочего дня

· Поля: ВСП (из vsp), дата, причина.
· Сохранение: INSERT INTO vsp_non_working_days (user_name, filial_id, vsp_id, date, reason).

Ранее добавленные нерабочие дни

· Запрос с JOIN к filials и vsp:

```sql
SELECT 
    nwd.id, nwd.user_name, f.name AS filial, v.name AS vsp,
    nwd.date, nwd.reason, nwd.created_at
FROM vsp_non_working_days nwd
JOIN filials f ON nwd.filial_id = f.id
JOIN vsp v ON nwd.vsp_id = v.id
WHERE nwd.filial_id = :filial_id [AND nwd.vsp_id = :vsp_id]
ORDER BY nwd.date DESC, nwd.created_at DESC
```

· Возможность фильтрации по конкретному ВСП или все (чекбокс «Показать все выходные вашего ВСП» управляет наличием условия vsp_id = :vsp_id).
· Удаление: DELETE FROM vsp_non_working_days WHERE id = :id AND vsp_id = :vsp_id AND filial_id = :filial_id.

Вкладка «Аналитика по филиалу»

Фильтры

· Филиал берётся автоматически из last_filial_id пользователя.
· Период дат (date_from, date_to).

Таблица всех проверок

· Используется запрос get_filial_sessions(filial_id) с дополнительной фильтрацией по дате.
· Поля аналогичны истории, но для всех пользователей филиала:
  · id, Сотрудник (s.user_name), Дата проверки, ВСП, Статус, Выполнено проверок, Всего проверок, Дата и время начала, Дата и время завершения.
  · Если реализован комментарий к сессии, добавляется s.completion_comment.

Управление черновиками

· Черновики = status_bul = false.
· Дополнительный фильтр по ВСП = last_vsp_name пользователя.
· Отображение: ID, Сотрудник, Дата, Прогресс.
· Массовое удаление: цикл по выбранным ID с вызовом delete_session.

---

Вкладка «Визуализация»

Статус ВСП на дату

· Источник: объединение данных из vsp (активные), vsp_non_working_days и checklist_sessions.
· Логика:
  1. Получить все активные ВСП филиала (vsp WHERE filial_id = :filial_id AND close_vsp = false).
  2. Получить множество ВСП, для которых есть нерабочий день на дату (SELECT vsp_id FROM vsp_non_working_days WHERE date = :date AND filial_id = :filial_id).
  3. Получить множество ВСП, для которых есть завершённая сессия на дату (SELECT vsp_id FROM checklist_sessions WHERE operation_date = :date AND filial_id = :filial_id AND status_bul = true).
  4. Для каждого ВСП:
